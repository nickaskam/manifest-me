Server Setup

Step 1 — Run the new database migration

Apply Subscription Model to Neon

Run this from your backend/ folder. Set DATABASE_URL to your Neon connection string so it targets production, not local SQLite.

cd backend
source venv/bin/activate

# Point at your Neon DB (same value as Cloud Run secret)
export DATABASE_URL="postgres://USER:PASSWORD@HOST/neondb?sslmode=require"

python manage.py migrate
  • This creates the Subscription table in Neon — required before deploying new API code
  • Safe to run multiple times (Django skips already-applied migrations)
Step 2 — Deploy to Cloud Run

Deploy API + Worker with new settings

The new deploy.sh sets the correct memory, CPU, and min-instance counts automatically.

cd backend

# Deploy both services
./deploy.sh

# Or deploy individually:
./deploy.sh api     # API only  (min 1 instance, 512MB)
./deploy.sh worker  # Worker only (4GB RAM, 2 CPUs, max 5 instances)
  • You must have gcloud CLI installed and authenticated (gcloud auth login)
  • Artifact Registry must have the manifest-me repo — create it if it doesn't exist:
    gcloud artifacts repositories create manifest-me --repository-format=docker --location=us-central1
  • After deploy, verify the health check responds: curl https://manifest-me-api-79704250837.us-central1.run.app/api/health/
Step 3 — App Store Connect

Create the Subscription Product

  • Go to App Store Connect → Your App → Subscriptions
  • Create a subscription group (e.g. "ManifestMe Premium")
  • Add a monthly product with ID exactly: com.manifestme.monthly
  • Set price (e.g. $4.99/mo), add localization, submit for review
  • In Xcode: add In-App Purchase capability under Signing & Capabilities
Step 4 — Xcode: Face ID permission

Add NSFaceIDUsageDescription

  • In Xcode, select the ManifestMe target → Info tab
  • Click + and add key: Privacy - Face ID Usage Description
  • Value: ManifestMe uses Face ID to keep your manifestations private.

Testing Plan

Test 1 — Health Check & Cold Start

Backend is warm and responding

curl https://manifest-me-api-79704250837.us-central1.run.app/api/health/
# Expected: {"status": "ok"} in under 2 seconds
  • Run it twice — second call should be just as fast (min-instances=1 keeps it warm)
  • If slow on first call, check Cloud Run console that min-instances was applied
Test 2 — Subscription Quota (Backend)

Manifest endpoint rejects unsubscribed users

# 1. Get a token (login)
TOKEN=$(curl -s -X POST \
  https://manifest-me-api-79704250837.us-central1.run.app/api/login/ \
  -H "Content-Type: application/json" \
  -d '{"username":"your@email.com","password":"yourpassword"}' \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['access'])")

# 2. Check subscription status
curl -H "Authorization: Bearer $TOKEN" \
  https://manifest-me-api-79704250837.us-central1.run.app/api/subscription/status/
# Expected: {"is_active": false, "videos_remaining": 0}

# 3. Try to manifest without a subscription
curl -X POST -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"prompt":"I am thriving","theme":"beach"}' \
  https://manifest-me-api-79704250837.us-central1.run.app/api/manifest/
# Expected: 403 {"error": "subscription_required"}
Test 3 — Session Fix (401 auto-refresh)

App recovers from an expired token without logging you out

  • Build and run the app in Xcode on a simulator or device
  • Log in normally — you should see the Gallery tab load your videos
  • Simulate expired token: In Xcode, pause the app, open the LLDB console at the bottom, and run:
    e KeychainHelper.standard.save(token: "expired.invalid.token")
    Then resume the app
  • Navigate to the Gallery tab (which calls fetchVideos)
  • Expected result: App silently refreshes the token using the stored refresh token and videos load normally — you are NOT logged out
  • If the refresh token is also expired (e.g. after many days), the app should show the Login screen cleanly
Test 4 — Onboarding

Shown once on first install, never again

  • Delete the app from the simulator/device
  • Build and install fresh from Xcode
  • Expected: 4-screen onboarding appears (Welcome → How It Works → Photo Permission → Get Started)
  • Tap through all screens and tap "Get Started" — you should land on the Login screen
  • Log in, use the app, then kill and reopen
  • Expected: Onboarding does NOT appear again — you go straight to the app
  • To reset onboarding during testing, run in LLDB: e UserDefaults.standard.set(false, forKey: "hasCompletedOnboarding")
Test 5 — Face ID

App locks on foreground if enabled

  • Log in on a physical device (Face ID only works on real hardware)
  • Go to Profile tab → toggle on "Require Face ID on open"
  • Face ID prompt appears immediately to confirm — authenticate
  • Press the Home button to background the app
  • Reopen the app from the dock
  • Expected: Black lock screen with Face ID icon appears, then automatically prompts Face ID
  • Authenticate successfully → lock screen dismisses, app is usable
  • Test failure: cover your face / decline — "Unlock with Face ID" button should appear to retry
  • Toggle off in Profile → background and reopen → no lock screen
Test 6 — StoreKit 2 Subscription (Sandbox)

Purchase flow, paywall, and quota

  • In Xcode: Product → Scheme → Edit Scheme → Run → Options → set StoreKit Configuration to your .storekit file (create one under File → New → StoreKit Configuration if needed — add the com.manifestme.monthly product there)
  • Build and run on simulator or device
  • Log in, go to Create tab, select a theme
  • Enter a prompt and tap "Start Manifesting"
  • Expected (unsubscribed): Paywall sheet slides up showing price and features
  • Tap "Subscribe" → StoreKit sandbox purchase dialog appears → confirm
  • Expected: Paywall dismisses, manifest starts, progress bar appears
  • Try to create a second video in the same month
  • Expected: Error toast "You've used your video for this month"
  • Test "Restore Purchases" button — should reactivate the subscription
Test 7 — Worker OOM (Cloud Run)

Video generation completes without crashing

  • Subscribe (or temporarily comment out the subscription check in manifest_video view for testing)
  • Submit a real manifest request from the app
  • Go to Google Cloud Console → Cloud Run → manifest-me-worker → Logs
  • Expected: Logs show [worker] START → CALLING VERTEX → DONE with no memory errors
  • If you see Memory limit exceeded or container exits with code 137, the 4GB setting did not apply — re-run ./deploy.sh worker and verify in the Cloud Run console under Configuration that Memory is set to 4Gi
